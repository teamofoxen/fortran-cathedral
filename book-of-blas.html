The Book of BLAS

BLAS — the Basic Linear Algebra Subprograms — is not a single program. It is a published specification of low-level vector and matrix operations, with a Fortran reference implementation kept at Netlib[1]. Its genius is the separation of interface from implementation: code written against the standard routine names can run on the plain reference Fortran, or on a machine-tuned library that implements the same interface many times faster[2]. That bargain — portable mathematics, replaceable speed — is why so much numerical software chose to stand on it.

The operations are organized in three levels, named in order of their arrival and their appetite: vector operations (Level 1, specified in 1979), matrix–vector operations (Level 2, 1988), and matrix–matrix operations (Level 3, 1990)[2][3]. Each exhibit below is computed, laid into its tables, and drawn into its map by this site's own Fortran at generation time.

Level 1: AXPY, the vector verse

Here the economics change. Multiplying an n×n pair performs O(n³) arithmetic on only O(n²) data — many operations per number moved. That surplus is what optimized implementations spend: blocks of the matrices can be staged in fast cache and reused, so a tuned GEMM approaches what the arithmetic units can actually sustain. This is why blocked algorithms put matrix–matrix work in their inner loops[6], and why Level 3 is the level modern hardware loves best.

BLAS and LAPACK

BLAS supplies the bricks; LAPACK builds the cathedral. LAPACK — the Linear Algebra PACKage, written in Fortran[4] — provides the higher operations of scientific computing: solving linear systems, least squares, eigenvalues, singular values. Its routines are written so that as much computation as possible is performed by calls to the BLAS, and it was designed at the outset to exploit Level 3[5]. Install a faster BLAS beneath it, and LAPACK gets faster without changing a line — the key to its portability[6].

The call chain

A user of a high-level environment may never see these names, yet may stand on them. One common shape of the descent:

  1. A researcher calls a high-level linear-algebra routine — NumPy's and SciPy's, for instance, are documented as relying on BLAS and LAPACK[7][8].
  2. The environment dispatches to LAPACK drivers for factorizations and solves[5].
  3. LAPACK casts the heavy work as BLAS calls, Level 3 wherever it can[5][6].
  4. The BLAS interface is served by whatever implementation is installed: the reference Fortran from Netlib[1], or an optimized library such as OpenBLAS[9] or Intel oneMKL[10].

Glossary

BLAS
Basic Linear Algebra Subprograms: a standardized set of low-level vector and matrix routines[1].
Reference implementation
The plain, portable Fortran realization of the BLAS kept at Netlib; correct everywhere, tuned for nowhere[2].
Optimized implementation
A library implementing the same interface with architecture-specific tuning, such as OpenBLAS or oneMKL[9][10].
Level 1 / 2 / 3
The operation classes: vector–vector, matrix–vector, and matrix–matrix[3].
AXPY, GEMV, GEMM
The archetypal routine of each level; a precision prefix (S, D, C, Z) completes the name[2].
LAPACK
The Linear Algebra PACKage: factorizations, solvers, and eigenproblems built on the BLAS[4][5].
Arithmetic intensity
Operations performed per datum moved; the resource Level 3 has in surplus and optimized libraries exploit[6].
Blocked algorithm
An algorithm reorganized to work on submatrices so its inner loops become Level 3 calls[6].
