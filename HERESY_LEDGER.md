# Heresy Ledger

This document records the Cathedral's heresy in two registers.

**Necessary heresies** are non-Fortran executable logic, or external
programs at genuine platform boundaries, that could not reasonably be
avoided. They are minimized, isolated, justified, and counted.

**Committed heresies** are avoidable violations: operational mistakes,
architectural lapses, residue admissions, or failures to grant Fortran
ownership it could plausibly have held. They are events, not code. They
are recorded permanently, verified by `forty confess`, displayed in the
public Confessional, and never alter the executable-heresy line count.
Correction changes the status of a committed heresy. It does not erase
it.

Forty parses, verifies, counts, and renders this ledger.

## Classification rules

Not heresy:

- HTML output,
- CSS,
- Markdown content,
- generated JSON, YAML, XML, SVG, or text,
- Fortran source,
- compiler and linker binaries,
- Git,
- the GitHub CLI,
- the browser,
- the hosting platform.

Potential executable heresy:

- JavaScript,
- TypeScript,
- Python,
- Ruby,
- handwritten C or C++ wrappers,
- shell scripts containing project logic,
- conventional web frameworks,
- external generators that perform application work.

## Necessary platform boundaries

External programs standing at genuine platform boundaries. Forty
orchestrates each one — the sequencing, the arguments, the capture, the
parsing, and the verdicts remain in Fortran — and none of them owns the
architecture.

| Boundary | Role | Why Fortran cannot absorb it |
|---|---|---|
| GFortran | Compiles the Cathedral; renders the compiler verdicts for the Hall's probe files | Forty cannot be his own compiler |
| fpm | Orders the build and the trials | The Fortran-native build path |
| Git | Keeps the vault; forms the commits and refs that Forty plans and verifies | Version control is a platform, not application logic |
| GitHub CLI | Holds credentials; creates the repository; appoints GitHub Pages | Secure credential handling must remain with the official tool |
| cmd.exe | The interpreter through which execute_command_line addresses Windows | The operating system's own doorway |
| curl | Fetches the public fabric for deploy verification and public inspection | The transport Windows already provides |
| Headless Chromium (Edge or Chrome) | The rendering engine that `forty inspect` measures through, invoked with its official debugging flags; the page-evaluation expressions the protocol requires are generated by Forty at run time into ignored ground, sent through the instrument, and parsed and judged by Forty — never tracked, never shipped | Layout facts exist only inside a rendering engine |
| GitHub Pages | Serves the porch to the public | Static hosting is a platform boundary |
| The visitor's browser | Renders what Fortran wrote; receives 0 lines of JavaScript | The web's own half of the contract |

## Rules

Every new executable impurity must be added to the executable register
in the same change that introduces it.

Every committed heresy must be recorded in this chapter as soon as it is
recognized, with its offense, consequence, commits, remediation, and
status. Correction updates the status and appends evidence. No entry is
ever deleted.

A vague justification such as “easier,” “standard,” or “best practice”
is insufficient.

The preferred removal path for executable impurity is always one of:

- replace with Fortran,
- have Forty generate it,
- reduce it to declarative data,
- or prove that it is a genuine platform boundary.
